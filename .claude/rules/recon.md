# VIPER Rules: Reconnaissance Best Practices

These rules govern VIPER's reconnaissance activities, ensuring efficiency, effectiveness, and strict adherence to ethical and legal boundaries.

## 1. Passive Reconnaissance (Always Preferred)

### 1.1 Definition
- **Rule**: Gather information about a target without directly interacting with it.
- **Examples**: OSINT (Open Source Intelligence), WHOIS lookups, DNS records, public code repositories (GitHub, GitLab), Shodan/Censys, Wayback Machine, Google Dorks.
- **Action**: Prioritize passive techniques to minimize detection risk and stay within scope without active interaction.

### 1.2 Data Sources
- **Rule**: Utilize a diverse set of public data sources for comprehensive information gathering.
- **Action**: Integrate with tools like Subfinder (for DNS/Certificate transparency), Gitrob (for public code), and custom OSINT scripts.

### 1.3 Scope Verification
- **Rule**: Passive recon should always be performed first to identify potential targets and verify their inclusion in the `scopes/current_scope.json`.
- **Action**: Any identified asset that is not explicitly in scope must be flagged and not further investigated.

## 2. Active Reconnaissance (Proceed with Caution)

### 2.1 Definition
- **Rule**: Directly interact with the target to gather information, such as port scanning, directory brute-forcing, and vulnerability scanning.
- **Requirement**: Active recon must only be performed on explicitly in-scope targets.
- **Action**: Use `viper_core.py` functions for active recon, which incorporate automatic scope validation.

### 2.2 Rate Limiting (Absolutely Critical)
- **Rule**: All active reconnaissance tools and scripts **MUST** adhere to strict rate limiting to avoid overwhelming target infrastructure or triggering WAF/IPS systems.
- **Guideline**: Implement delays between requests, use slower scanning speeds, and configure tools (e.g., Nuclei, Nmap) with `rate-limit` options.
- **Action**: Use dynamic rate limiting algorithms based on target responsiveness and `robots.txt` directives.

### 2.3 User-Agent String
- **Rule**: Use a custom, identifiable User-Agent string for all active reconnaissance requests.
- **Example**: `User-Agent: VIPER Bug Bounty Scanner (Authorized Testing by viper-ashborn)`
- **Action**: Configure `http_client.py` and other tools to use this User-Agent. This helps target owners identify legitimate testing.

### 2.4 No Destructive Payloads
- **Rule**: During active recon (e.g., fuzzing, parameter discovery), never use payloads that could lead to data modification, deletion, or denial of service.
- **Action**: All payloads generated by `payload_mutator.py` are designed to be read-only and non-impactful during recon.

### 2.5 Time-of-Day Considerations
- **Rule**: For more aggressive active reconnaissance, consider performing activities during off-peak hours for the target (if program rules allow).
- **Action**: Automated scheduling can be configured in `settings.local.json`.

## 3. Scope Expansion & Validation

### 3.1 Discovering New Assets
- **Rule**: New subdomains, IP addresses, or applications discovered during recon should be cross-referenced against the program's scope.
- **Action**: If a new asset appears to be in-scope but isn't explicitly listed, it should be validated and, if appropriate, added to `scopes/current_scope.json`.

### 3.2 Out-of-Scope Discovery
- **Rule**: If a clearly out-of-scope asset is discovered (e.g., third-party service, internal network component), immediately cease investigation of that asset.
- **Action**: Document the discovery without further probing and mark it as out-of-scope.

## 4. Automation and Tooling

### 4.1 Approved Tools
- **Rule**: Only use approved and configured reconnaissance tools (e.g., Subfinder, Amass, HttpX, Nuclei for information gathering).
- **Action**: Integrate new tools via the `tools/` directory with proper configuration and safety checks.

### 4.2 Logging
- **Rule**: Maintain detailed logs of all reconnaissance activities, including targets, tools used, timestamps, and findings.
- **Action**: All recon module outputs are directed to `logs/recon_logs/` for auditability.

### 4.3 Data Storage
- **Rule**: Store reconnaissance data securely and locally. Do not upload raw recon data to public repositories.
- **Action**: Processed and filtered data can be used for further analysis but raw data remains sensitive.

## 5. Continuous Reconnaissance

### 5.1 Monitoring Changes
- **Rule**: Periodically re-run passive reconnaissance to detect changes in the target's attack surface (new subdomains, updated DNS records, new technologies).
- **Action**: Configure `viper_daemon.py` for scheduled, continuous recon tasks.

### 5.2 Dynamic Scope Updates
- **Rule**: Automatically update the internal representation of the target's attack surface based on new recon findings.
- **Action**: Integrate with the `scopes/` management system to ensure an up-to-date view of the target.

---

*Reconnaissance is the foundation of successful bug bounty hunting. These rules ensure a thorough yet ethical approach.*
