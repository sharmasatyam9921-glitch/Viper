# PLAN — Function Audit: `memory`

> Part of [PLAN.md](PLAN.md) Section 2. Generated 2026-06-06. Module: **memory/viper_memory.json + handlers: archive/old_agents/agentic_viper.py (Memory class, writer/reader) and core/viper_db.py (migrate_from_json, one-time reader)**

**Exists:** True  

## Summary

viper_memory.json (9.8 KB) is a flat JSON store of attack outcomes. SHAPE: top-level dict with keys successful_attacks (list of {target, technique, payload, result, timestamp} plus optional CTF fields flag/challenge/points/difficulty), failed_attacks (list of {target, technique, payload, error, timestamp}), learned_patterns ({}), target_history ({}), techniques ({}), session_count (int=96), last_save (ISO ts), and hackerone_hunts (list of program-hunt summaries). WRITER: the Memory class in archive/old_agents/agentic_viper.py (save / remember_success / remember_failure / learn_pattern + session_count bump in AgenticVIPER.__init__). It is the ONLY thing that writes this file. That class lives under archive/ and is imported only by other archived agents (smart_viper.py, learned_capabilities.py, viper_agentic_run.py, viper_learn.py) — nothing in the live pipeline. READER: same Memory._load() reads it back; additionally core/viper_db.py migrate_from_json() reads it ONCE for a SQLite migration, but that method is invoked only from viper_db.py's __main__ block, not from any live hunt path. So in the current/active VIPER 5.0 codebase this file is effectively dead/legacy data that only gets touched if someone runs the archived agents or `python core/viper_db.py` manually. TTL/SIZE/ROTATION: no TTL and no rotation by time; the only bound is a hard slice — successful_attacks kept to last 500, failed_attacks to last 200 (in-memory before write). learned_patterns/target_history/techniques/hackerone_hunts are UNBOUNDED (grow forever). No file-size cap, no compaction, no backup/atomic write.

## Top issues

- WRITE PATH IS NON-ATOMIC + WRITE-PER-EVENT: Memory.save (agentic_viper.py:65-67) does write_text of the whole file on every remember_success/remember_failure/learn_pattern call. No tmp-file+rename, so a crash or two concurrent agent runs corrupt the file; combined with O(n) full rewrite per event this is a race + performance hazard.
- SILENT DATA-LOSS ON CORRUPT FILE: Memory._load (agentic_viper.py:54-55) uses `except Exception: pass` and silently returns an empty default skeleton, discarding all on-disk history without any log if the JSON is malformed.
- NO IDEMPOTENCY / DUP EXPLOSION: neither remember_failure (agentic_viper.py:81) nor migrate_from_json (viper_db.py:305) dedups. The live file already shows ~18 identical natas2 failure rows, and re-running migrate_from_json re-inserts every target/attack into SQLite.
- UNBOUNDED GROWTH: only successful_attacks(500)/failed_attacks(200) are trimmed; learned_patterns, target_history, techniques, and hackerone_hunts have no TTL, size cap, or rotation — they grow forever. There is no time-based TTL anywhere.
- SECRETS PERSISTED UNREDACTED: remember_success stores raw result/payload to disk; viper_memory.json on disk contains real CTF flags (HTB{...}) and recovered natas passwords in cleartext, violating the project's 'findings redacted / no credentials' rule.
- DEAD/STALE MODULE: the only writer (Memory class) lives in archive/old_agents/ and is referenced only by other archived agents; migrate_from_json is reachable only via `python core/viper_db.py` __main__. The live VIPER 5.0 pipeline neither reads nor writes this file, so it is stale legacy state.

## Scope / safety notes

The Memory handler itself (agentic_viper.py Memory class) performs NO network/target action — it is pure local-file JSON I/O, so no scope/RoE check is needed there. However the writer's host module AgenticVIPER.hack() (agentic_viper.py:500) DOES perform live network actions (Tools.http_get/http_post to the target, plus active LFI/SQLi/CMDi/cookie/header-spoof attacks at lines 360-470) with NO scope or RoE check anywhere in this file — it hits whatever URL is passed on argv. That is a scope-enforcement gap, but it is in archived/dead code outside the live pipeline (live VIPER routes scope through roe_engine.py/guardrails.py). migrate_from_json (viper_db.py:305) is local DB-only, no network. Net: the memory file's read/write code is network-free; the legacy agent that feeds it is not scope-gated, but it is archived.
