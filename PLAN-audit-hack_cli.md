# PLAN — Function Audit: `hack_cli`

> Part of [PLAN.md](PLAN.md) Section 2. Generated 2026-06-06. Module: **hack_cli (core/hack_cli.py + core/hack_profile.py)**

**Exists:** True  

## Summary

Two files: hack_cli.py is the argparse entrypoint for `viper.py hack <target>` — it parses flags, runs preflight checks, builds a Profile + Narrator + ScopeReasoner + AuditLogger, then delegates the actual hunt to HackMode.run() (async, via asyncio.run) and writes a summary JSON. hack_profile.py defines the Profile dataclass + StopCondition, three factory profiles (CTF/BugBounty/Lab), stop-condition predicates, and detect_profile() auto-detection. The CLI performs NO direct network/target action itself — all probing and scope enforcement are delegated to HackMode/ScopeReasoner downstream. Code is clean and well-documented; cross-references to HackMode (resume/run, HackResult fields, AuditLogger.for_hunt/event/close) all verified present. Main concerns: a few broad excepts (logged, not silent), one silent `except Exception: continue` in should_stop that can swallow a faulty stop-condition (max_iterations backstops it), relative default paths, and the security-relevant fact that scope-reasoner build failure degrades to scope_reasoner=None with only a WARN (default-deny must be honored downstream, not enforced here).

## Top issues

- detect_profile (hack_profile.py:286,302-305) forces allow_destructive=True AND use_scope_reasoner=False for CTF-pattern hostnames and private IPs regardless of the --go flag — a target matching the CTF regex (e.g. attacker-controlled 'ctf.victim.com') silently enables destructive exploit/post workers with no scope rails by default.
- Fail-open scope handling: when the scope file fails to load or the reasoner build raises, hack_cli (lines 166-167, 253-256) prints only a [WARN] and proceeds with scope_reasoner=None; the documented default-deny is only as strong as HackMode's handling of a None reasoner (unverified).
- Profile.should_stop (hack_profile.py:145) has a silent `except Exception: continue` that swallows errors from a faulty stop-condition; a perpetually-raising condition would never fire, leaving only max_iterations to backstop an otherwise unbounded persistence loop.
- Documented exit codes 2 (guardrail blocked) and 3 (approval denied) are never returned by run_hack_cli — guardrail/approval outcomes are not surfaced to the shell exit code, so callers/scripts cannot distinguish them.
- Minor: --report and --no-dashboard flags are parsed but never consumed in run_hack_cli; list_profiles (hack_profile.py:320) is dead code; default paths state/hunts and data/viper.db are relative (CWD-dependent).

## Scope / safety notes

hack_cli.py performs NO direct network or target action — it does not open sockets, send HTTP, or scan. It only parses args, validates the target string / scope-file existence (Path.exists), builds Profile/Narrator/ScopeReasoner/AuditLogger objects, then delegates the entire hunt to HackMode.run() via asyncio.run. Scope/RoE enforcement is therefore delegated downstream to ScopeReasoner + HackMode, not done here. Two safety gaps to flag: (1) if the scope file fails to load or the reasoner build raises, the CLI prints a [WARN] and continues with scope_reasoner=None (hack_cli.py:166-167, 253-256) — fail-open at this layer; the 'default-deny' guarantee is only as strong as HackMode's handling of a None reasoner. (2) detect_profile auto-selects CTFProfile/LabProfile with use_scope_reasoner=False AND allow_destructive=True for CTF-pattern hostnames and private IPs, independent of the --go flag (hack_profile.py:286,302-305) — meaning a target that matches the CTF regex or is a private IP runs with no scope reasoner and destructive workers enabled by default. The CLI itself never cross-checks the resolved target against the scope file before launching. No secrets are logged; db_path/scope paths are the only sensitive-ish values printed and they are user-supplied config, not credentials.
